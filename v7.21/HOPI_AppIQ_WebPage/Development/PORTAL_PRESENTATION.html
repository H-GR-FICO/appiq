Filozofie platformy — Původní záměr Platform Philosophy — Original Intent
Proč vznikla HOPI AppIQ? Why does HOPI AppIQ exist?
"Build smart. Deploy fast. Scale everywhere."

Původní záměr: Hledáme platformu pro rychlý vývoj, nasazení a distribuci aplikací vyvíjených firmou a jejími uživateli za pomoci umělé inteligence. Tento záměr je kostra, která prostupuje celou architekturou — každé rozhodnutí se hodnotí prizmatem: „Umožňuje toto naší firmě rychleji stavět, nasazovat a sdílet aplikace s AI?" Original Intent: We are looking for a platform for rapid development, deployment and distribution of applications built by the company and its users with the help of artificial intelligence. This intent is the backbone running through the entire architecture — every decision is evaluated through the lens: "Does this enable our company to build, deploy and share applications with AI faster?"

1
Tvorba Build
Každý člen organizace je potenciálním tvůrcem aplikace. HOPIQ snižuje bariéru vstupu na minimum — analytik navrhne aplikaci bez znalosti programování. AI Configurator převede zadání na funkční modul. Every member of the organization is a potential app creator. HOPIQ lowers the barrier to a minimum — an analyst can design an application without programming knowledge. AI Configurator converts requirements into a functional module.
2
Nasazení Deploy
Platforma se postará o zbytek automaticky. CI/CD pipeline od kódu k produkci bez manuálního zásahu. Azure Static Web Apps zajistí hosting, certifikáty a škálování. Entra ID spravuje přístup přes existující M365 účty. The platform handles the rest automatically. CI/CD pipeline from code to production without manual intervention. Azure Static Web Apps provides hosting, certificates and scaling. Entra ID manages access via existing M365 accounts.
3
Škálování Scale
Jednou postavené jádro slouží celé organizaci. Finance (první deployment) → Operations → Purchasing → HR → IT. Každá oblast dostane svůj sub-portál HOPI AppIQ | [Oblast]. Bez přepisu kódu. Bez vendor lock-in. Core built once serves the entire organization. Finance (first deployment) → Operations → Purchasing → HR → IT. Each area gets its sub-portal HOPI AppIQ | [Area]. No code rewrite. No vendor lock-in.

Výhled — Skalování mimo HOPI

HOPI AppIQ je navržen jako nasaditelný organizační inteligentní model — tato vlastnost platí nejenom uvnitř skupiny HOPI, ale i pro nasazení do jiných organizací, white-label provoz nebo prodej na trhu jako SaaS produkt. Klíčové rozhodnutí: Architektonické základy pro multi-tenant provoz zabudovat nyní (nulový náklad při novém přepisu), ne zpětně (prohibitivní náklad refaktorizace). Samotné SaaS funkce se nestaví v Fázi 1 — ale nic je nezablokuje. Key decision: Build the architectural foundations for multi-tenant operation now (zero cost in a fresh rewrite), not retroactively (prohibitive refactoring cost). The actual SaaS features are not built in Phase 1 — but nothing will block them.

🏢

Varianta A — Interní produktizace

Nasazení stejného portalu na sesterskou firmu, dceřinou organizaci nebo skupinu s podobnou strukturou. Nový tenant sdílí jádro platformy, má vlastní Admin Config, vlastní moduly, vlastní branding. Deploying the same portal to a sister company, subsidiary or group with a similar structure. The new tenant shares the platform core, has its own Admin Config, own modules, own branding.

Architektonický dopad: Minimální — přidání tenant.* vrstvy do AppState. 4D model to zvládá přirozeně. Architectural impact: Minimal — adding the tenant.* layer to AppState. The 4D model handles this naturally.

Varianta B — SaaS / Trh

Pronájem platformy cizím firmám jako produkt. Každý klient = izolovaný tenant. White-label branding, vlastní doména, vlastní uživatelé a oprávnění. Plná datová izolace. Renting the platform to external companies as a product. Each client = isolated tenant. White-label branding, custom domain, own users and permissions. Full data isolation.

Architektonický dopad: Zásadnější — billing, data sovereignty, white-label, support tiers. Vyžaduje byznys rozhodnutí. Architectural impact: More significant — billing, data sovereignty, white-label, support tiers. Requires a business decision.

The tenant layer is an optional root configuration — it sits between Factory Defaults and Admin Config. Each tenant has its own isolated namespace. Ignored for internal HOPI deployments.
